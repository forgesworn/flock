# App lock — key-at-rest (Phase E via keystore-kit, closes the Phase G caveat)

`flock:v1` has been a plaintext localStorage blob holding the identity secret,
the circle root, every circle seed, petnames, and private places — the "not
secure key storage (MVP only)" caveat flagged in README/CLAUDE.md since the
start. The app lock closes it: an opt-in PIN puts the **whole persisted state**
behind AES-256-GCM at rest.

## Decisions

### keystore-kit, vendored until published

The kit is the toolkit's key-at-rest primitive (extracted from the audited
signet-app; PBKDF2-600k → AES-GCM; PIN / WebAuthn-PRF / grace; `burn()`), but
it is not yet on npm and its repo is private — so flock consumes a packed
tarball (`vendor/keystore-kit-0.1.0.tgz`, `file:` dep — works in CI). Swap to
a version range once it's published (`npm login && npm publish` in the kit).

### The shape: a random storage secret, PIN-wrapped

- Enabling generates a random 256-bit **storage secret** (`generateSecret`).
- The secret is PIN-wrapped by keystore-kit (`setupPIN`) under namespace
  `flockks`.
- Every save thereafter writes `{locked: 1, d: <canary-kit envelope>}` —
  the same envelope crypto as every other sealed thing in flock.
- Boot sees the `locked` marker → a PIN screen *before* any state exists in
  memory; `unlockPIN` recovers the secret, decrypts, and boots normally.

### Grace: no PIN on every reload — but flock owns the window

keystore-kit's grace wraps the secret under a **non-extractable** CryptoKey in
IndexedDB, so a page reload can unlock silently — but the kit has no TTL, so
flock keeps a `flockks.graceUntil` timestamp (15 min): within it, boot
unlocks via grace with no prompt; past it, the grace key is cleared and the
PIN screen shows. A timestamp key is not sensitive (the state blob is what's
protected).

### The async-save problem: a coalescing write queue

`store.save()` is synchronous and called everywhere; WebCrypto is async. The
store keeps the sync signature and, when the lock is armed, queues the
serialised state for an encrypt-then-write drain — rapid saves coalesce
(last-write-wins), and the drain re-checks the kill switches after each await
so neither a hide (`lockSaves`) nor a disable can be raced into resurrecting
or clobbering state. Belt-and-braces guard: with no key armed, a save while
the at-rest blob is ciphertext is a **no-op** — a stray pre-unlock save must
never overwrite the ciphertext with fresh plaintext.

### Composing with the decoy view

- The storage secret also lives **inside** `Persisted.lock` — inside the very
  blob it encrypts, which is cryptographically free (you can't read it without
  already having decrypted) and exactly the decoy-key pattern.
- **Hide** seals the in-memory plaintext (secret included) into `flock:cache`
  and leaves the keystore blobs alone — the decoy boots with no PIN screen
  (lock detection keys off the `locked` marker in `flock:v1`, which is gone),
  and a leftover `flockks.*` blob adds nothing beyond the already-documented
  `flock:cache` forensic limit.
- **Unhide** restores plaintext. The app boots unlocked, detects
  `lock.secret` present but plaintext at rest, and shows a **re-confirm
  card**: enter the PIN once → `setupPIN` re-wraps *our* secret (healing even
  the absurd corner where someone set their own lock inside the decoy) → the
  state re-encrypts. Saves stay plaintext until re-confirmed — deliberate,
  visible, one tap from healed.
- **Restore from backup** never carries the lock (`Persisted.lock` is
  device-specific, excluded like the relay list) — a restored device starts
  unlocked and enables afresh, so "plaintext + lock armed" uniquely means
  "just unhidden".

### Escape hatch: a forgotten PIN must not be a brick

The lock screen carries a two-step "I've forgotten my PIN" → wipe this device
(state + keystore burn) → onboarding → restore from backup. Same recovery
story as a lost phone, stated on the enable card: **make a backup first**.

## Honest limits

- A PIN is entered on this device: it resists a snoop, a lifted phone, a
  browser-profile copy — not a determined offline attacker doing PBKDF2 runs
  against a 4-digit PIN. The card copy nudges toward 6+ characters.
- WebAuthn-PRF biometric unlock is wired in the kit but not surfaced yet —
  needs real hardware to validate (same gate as everything Tier-2).
- While unlocked, state is in memory and the storage secret is live — the
  lock is at-rest protection, not runtime isolation.

## Implementation

- `app/src/store.ts` — `armRest`/`disarmRest`/`lockedAtRest`/`openRest`/
  `flushRest`/`sealOff`, the coalescing drain, `hydrate()` shared by
  `load()`/`openRest()`. Unit-tested with a stub localStorage.
- `app/src/lock.ts` — the keystore-kit wiring (browser adapters, namespace,
  grace window helpers). Browser-only; exercised by the e2e.
- `app/src/app.ts` — boot split (locked → PIN screen → unlock → normal boot),
  You→Advanced "App lock" card (enable / disable / re-confirm-after-unhide),
  `resetDevice` burns the keystore.
- E2e (`e2e/lock.spec.ts`) — ciphertext at rest; PIN gate after grace expiry
  (wrong rejected, right unlocks, B's live traffic decrypts after); grace
  reload skips the PIN; the full lock × decoy composition.
